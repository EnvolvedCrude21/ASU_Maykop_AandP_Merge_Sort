## Описание проекта

Данный проект реализует гибридный параллельный алгоритм сортировки слиянием (Parallel Merge Sort) на C++20. Алгоритм сочетает многопоточность с интеллектуальными оптимизациями для достижения максимальной производительности на современных многоядерных процессорах.

### Ключевые особенности

- Порог отсечения (Threshold): Автоматическое переключение на std::sort для малых массивов (16 384 элемента)
- Адаптивная многопоточность: Использование std::thread::hardware_concurrency() для определения оптимального количества потоков
- Zero-Allocation архитектура: Единый переиспользуемый буфер для операций слияния
- Оптимизация пропуска слияния: Пропуск операции merge, если половины уже отсортированы
- Визуализация в реальном времени: Интерактивная демонстрация работы алгоритма на SFML

## Архитектура и оптимизации

### 1. Порог отсечения (Threshold = 16 384)

```cpp
if (right - left <= threshold_) {
    std::sort(arr.begin() + left, arr.begin() + right + 1);
    return;
}
```

**Почему 16 384?**

- 16 384 элемента × 4 байта (int) = 65 536 байт = 64 КБ
- Это идеально помещается в L2-кэш процессора (256 КБ - 1 МБ)
- Когда данные в кэше, доступ к ним в 20-100 раз быстрее, чем к RAM

**Иерархия кэша процессора:**

| Уровень | Размер | Скорость доступа |
|---------|--------|------------------|
| L1 Data | 32-48 КБ | ~1 нс |
| L2 | 256 КБ - 1 МБ | ~3-4 нс |
| L3 | 8-32 МБ (общий) | ~10-15 нс |
| RAM | Гигабайты | ~80-100 нс |

**Избегание overhead:**

- Создание потока через std::async стоит ~10-50 микросекунд
- Сортировка 16 384 int через std::sort: ~50-100 мкс
- Сортировка 1 000 int через std::sort: ~3-5 мкс
- Если поставить threshold = 1000, время создания потока станет в 10 раз больше времени сортировки

**Формула для подбора:**

```cpp
threshold_ ≈ (Размер_L2_кэша_в_байтах / sizeof(T)) * 0.7
```

### 2. Контроль глубины рекурсии

```cpp
unsigned int num_threads = std::thread::hardware_concurrency();
int max_depth = 0;
while ((1 << max_depth) < num_threads) {
    max_depth++;
}
```

**Проблема без контроля:**

- Для массива из 10 млн элементов создадим ~10 млн потоков
- Каждый поток требует 1-8 МБ стека
- 10 000 000 потоков × 2 МБ = 20 Терабайт RAM → OOM Crash
- ОС тратила бы 99.9% времени на Context Switch

**Решение:**

- hardware_concurrency() возвращает количество логических ядер (например, 8)
- Битовый сдвиг 1 << depth — это возведение двойки в степень
- Для 8 ядер: 1<<0=1, 1<<1=2, 1<<2=4, 1<<3=8 → max_depth = 3
- Гарантирует создание ровно столько потоков, сколько у нас ядер

**Адаптивность:**

- На слабом ноутбуке с 2 ядрами: max_depth = 1 (2 потока)
- На сервере с 64 ядрами: max_depth = 6 (64 потока)
- Код автоматически подстраивается под любое железо

### 3. Zero-Allocation архитектура (ООП-подход)

**Проблема процедурного подхода:**

```cpp
void merge(...) {
    std::vector<T> L(...); // Создаем временный вектор
    std::vector<T> R(...); // Создаем еще один
}
```

- При сортировке 10 млн элементов создаем и уничтожаем векторы МИЛЛИОНЫ раз
- Миллионы операций new/delete → огромные накладные расходы
- Потоки конкурируют за аллокатор памяти (Memory Contention)

**Решение в ООП:**

```cpp
class ParallelMergeSorter {
private:
    std::vector<T> buffer_; // Буфер создается ОДИН раз
    
public:
    void sort(std::vector<T>& arr) {
        if (buffer_.size() < arr.size()) {
            buffer_.resize(arr.size());
        }
        sort_impl(arr, 0, arr.size() - 1, max_depth_);
    }
};
```

**Преимущества:**

- Буфер создается один раз при первом вызове sort()
- При последующих вызовах переиспользуется уже выделенная память
- Если отсортировать 100 массивов, память выделится только один раз

### 4. Оптимизация пропуска слияния

```cpp
if (arr[mid] <= arr[mid + 1]) return;
merge(arr, left, mid, right);
```

**Логика:**

- arr[mid] — последний (самый большой) элемент левой половины
- arr[mid + 1] — первый (самый маленький) элемент правой половины
- Если самый большой слева <= самого маленького справа, значит ВСЁ уже отсортировано
- Не нужно тратить время на тяжелую операцию merge

**Когда срабатывает:**

- Частично отсортированные данные (логи по времени, температурные данные)
- Данные с естественной упорядоченностью (цены акций, временные метки)
- После нескольких уровней рекурсии

**Эффект:**

- На реальных данных может пропустить 30-50% всех операций merge
- Экономит миллионы операций с памятью

### 5. Асимметричное ветвление

```cpp
auto left_future = std::async(...); // Левая половина → новый поток
sort_impl(arr, mid + 1, right, ...); // Правая половина → текущий поток
left_future.get(); // Ждем помощника
```

**Почему не два потока?**

- Если бы создали два потока, текущий поток остался бы без работы
- Он бы просто стоял и ждал, пока оба закончат → расточительство
- Вместо этого: один поток берет левую половину, текущий сам сортирует правую
- Утилизация ядер на 100% без простоев

## Технические детали

### Используемые возможности C++20

- std::async и std::future для асинхронного выполнения
- std::thread::hardware_concurrency() для адаптации под железо
- std::vector для хранения данных и буфера
- std::sort для сортировки малых кусков (Introsort)
- std::chrono для точных замеров времени
- Шаблоны для поддержки любых типов данных

### Сложность алгоритма

- Временная сложность: O(n log n) в среднем и худшем случае
- Пространственная сложность: O(n) для буфера
- Параллельная сложность: O(n log n / p), где p — количество потоков

### Ограничения

- Memory Wall: На больших объемах данных скорость ограничена пропускной способностью шины памяти
- Закон Амдала: Финальное слияние выполняется последовательно, что ограничивает максимальное ускорение
- Overhead потоков: На малых объемах данных (< threshold) многопоточность не дает преимущества
